# Server Clean Architecture + DDD

`CLAUDE.md` のサーバーアーキテクチャ章で示されたレイヤー構造に基づく **設計規約** を定める。dependency-cruiser が検証する import の依存方向以外の規約を扱う。

## コントローラの責務

コントローラ (`**/adapters/controllers/`) はリクエストの **受け取り・バリデーション・ユースケース呼び出し・レスポンス変換のみ** を行う。以下が含まれていたら違反:

- 条件分岐によるビジネスルール判定
- 複数ユースケースの組み合わせによるオーケストレーション
- エンティティの直接操作

ビジネスルールは **ユースケース層** に置き、コントローラは薄く保つ。

## ゲートウェイの責務

ゲートウェイ (`**/adapters/gateways/`) は domain 層のリポジトリインターフェースを実装し、永続化層との変換に集中する。以下を守る:

- ドメインオブジェクトと永続化スキーマの相互変換 (`toEntity` / `toRow`) のみ行う
- ビジネスロジック・条件分岐・複数操作の組み合わせは含まない（それらは usecase 層）
- インフラ起因の例外は **原則として `InfrastructureError` でラップする**。ただしアプリケーション層で分岐・認識される必要があるエラーは、後述「ポート専用エラーと層境界」に従い専用エラー型をスローする

## ドメインモデルの設計

エンティティ (`**/domain/*.ts`) は **データの入れ物だけにしない**。以下を保つ:

- ビジネスルール・状態遷移はエンティティのメソッドで表現する
- 値オブジェクトはコンストラクタでの検証だけでなく、必要に応じてドメイン操作を持つ
- ファクトリは `create` (新規生成) / `reconstruct` (永続化からの復元) を区別する

「貧血ドメインモデル」（メソッドが `create` / `reconstruct` だけで、ビジネスルールがすべてユースケース側にある状態）は避ける。

## プリミティブ執着の回避

ドメイン層では、**ドメイン上の意味を持つ値** をプリミティブ型 (`string`, `number`) のまま表現しない。以下は値オブジェクト化する:

- ID（例: `ProjectId`, `DocumentId`）
- ドメイン上の意味を持つ値（例: `ProjectName`, `DocumentFileSize`）

ユースケース・エンティティ間では、ID などのドメイン概念を `string` / `number` で直接やり取りせず、値オブジェクトで型保護する。技術的な値まで一律に値オブジェクト化することは求めない。

## ユースケースの責務

1 つのユースケースは **単一の責務** を持つ:

- 異なる集約を跨いだ操作を 1 メソッドで行わない
- ファイルが長すぎる場合（目安: 50 行超）は責務超過の疑い

引数は Command / Query オブジェクトで受ける（CQRS 流儀; 状態変更系は `XxxCommand`、読み取り系は `XxxQuery`）。

詳細は `server-application-result.md` (Result 型) と `server-hono-routes.md` (Hono ルート定義) を参照。

## ポート専用エラーと層境界

インフラ起因の例外は原則として `InfrastructureError` でラップするが、**アプリケーション層で認識・分岐されるべきエラー**（例: ファイル未発見、楽観ロック競合）は専用のエラー型を設け、それをスローする。

専用エラー型は **ポートが定義されている位置に置く**:

- ポートが `**/shared-kernel/` にある場合 → `**/shared-kernel/` に専用エラー型を定義
- ポートが各モジュールの `**/domain/` または `**/application/` にある場合 → 同じ位置に専用エラー型を定義

ポート位置を超えて内側（より抽象度の高い層）の汎用エラー型を infrastructure 実装が独自に直接構築・送出するのは層境界違反。ただしポート契約としてその汎用エラー型（例: shared-kernel ポートでの `DomainError`）が **明示的に採用されている** 場合は、契約に従った構築・送出を許容する。
